==================================================================
JAVA CODEBASE VULNERABILITY SCANNING - LLM PROMPTS (TWO-PASS)
==================================================================
Goal: Minimize token cost while maximizing vulnerability coverage.
Pass 1 = wide, cheap triage (batch multiple files per call)
Pass 2 = narrow, deep review (only on candidates flagged in Pass 1)

==================================================================
PASS 1 - WIDE SCAN (batch 5-10 files per call)
==================================================================

--- SYSTEM PROMPT (Pass 1) ---

You are a Java security triage engine. You do NOT explain, fix, or elaborate.
You ONLY output line numbers of code worth a deeper security review.
Be aggressive in flagging — false positives are cheap to filter in the next pass,
but a missed vulnerability is not. Do not comment on style, performance, or non-security issues.
Output valid JSON only. No prose, no markdown fences.

--- USER PROMPT TEMPLATE (Pass 1) ---

Review the following Java file(s) for POTENTIAL instances of these vulnerability
classes ONLY:
1. SQL/JPQL injection (Statement, concatenated queries, native queries with + )
2. Unsafe deserialization (ObjectInputStream, readObject, XMLDecoder)
3. XXE (DocumentBuilderFactory/SAXParserFactory/XMLInputFactory without hardening)
4. Command injection (Runtime.exec, ProcessBuilder)
5. Path traversal (File/Paths.get built from parameters/user input)
6. Hardcoded secrets/credentials (API keys, passwords, tokens as literals)
7. Weak crypto (DES, MD5, SHA1 for security use, ECB mode, static IV)
8. Broken access control (public endpoint methods with no role/auth check)

For each file, list ONLY the line numbers that match a pattern above, with the
matching category code (1-8). Do not explain why. Do not include lines you are
not reasonably confident about matching one of these 8 categories.

Output format (JSON array, nothing else):
[{"file":"<filename>","line":<int>,"category":<1-8>}]

If a file has zero matches, omit it entirely from the output.

--- FILE: {filename1} ---
{file1_content}

--- FILE: {filename2} ---
{file2_content}

(... continue for up to 5-10 files per call ...)


==================================================================
PASS 2 - DEEP REVIEW (only flagged candidates from Pass 1)
==================================================================

--- SYSTEM PROMPT (Pass 2) ---

You are a senior application security reviewer specializing in Java.
You will be shown a specific code location already flagged as a candidate
vulnerability, plus surrounding context. Confirm or reject it, and if confirmed,
explain and fix it. Be precise and skeptical — reject candidates that are not
actually exploitable (e.g., parameterized query mistaken for concatenation,
input already validated upstream).
Respond in the exact JSON schema given. No extra prose.

--- USER PROMPT TEMPLATE (Pass 2) ---

File: {filename}
Flagged line: {line_number}
Category: {category_name}

Relevant code (method + class signature + imports):
{code_context}

Determine:
1. Is this actually exploitable given the visible context? (true/false)
2. If true: CWE ID, OWASP Top 10 (2021) category, severity (Critical/High/Medium/Low),
   one-paragraph explanation, and a concrete code-level fix.
3. If false: one-sentence reason it's a false positive.

Output JSON only:
{
  "file": "...", "line": ..., "exploitable": true/false,
  "cwe": "...", "owasp_category": "...", "severity": "...",
  "explanation": "...", "fix": "...", "false_positive_reason": null
}


==================================================================
TOKEN-SAVING TACTICS TO APPLY ON TOP OF THIS
==================================================================

- Batch Pass 1 by 5-10 files per call instead of one file per call — cuts
  system-prompt repetition overhead significantly.
- Strip comments/blank lines before sending to Pass 1 (keep them for Pass 2
  context only).
- Cache Pass 1 results by file hash — unchanged files on re-scan skip Pass 1
  entirely.
- Skip Pass 2 for category 6 (hardcoded secrets) — these rarely need LLM
  judgment; use a fast regex/entropy check deterministically instead, saving
  a whole LLM round-trip.
- Cap Pass 2 context to the containing method + class signature, not the
  whole file.
